# 半导体研究 Agent 全链路验证

日期：2026-07-16

## 验收结论

半导体研究助手已连通“规划 → 检索 → 按需分析 → 写作 → 审核 → 修订/补充检索”的完整 Agent 链路。当前系统能够用本地小模型和指定的 Milvus 半导体知识库完成研究任务，并在证据不足时拒绝把报告判定为可发布，而是进入补充检索流程。

这轮验证证明的是 Agent 工作流与证据约束有效，不代表当前知识库已覆盖所有半导体细分问题。

## 真实链路结果

- 研究问题：Chiplet 互连接口标准、互操作性和测试验证的关键技术缺口。
- Architect 生成 3 个问题相关章节：证据与现状、关键缺口、工程建议与待验证项。
- Scout 从 `semiconductor_packaging_testing` 真实集合检索并整理 18 条事实。
- 本次任务没有可靠结构化数据和图表需求，DataAnalyst 与 CodeWizard 被条件路由跳过，生成图表数为 0。
- Writer 只允许保留实际检索结果中的 URL，虚构外链和虚构图表会被删除。
- 引用按照 URL 去重，同一文档的多个切片不会形成重复参考文献。
- Critic 发现核心论断缺少直接证据后，将质量分限制为 4 分，并把流程路由到 `re_researching`，没有放行报告。

## 本轮修复

1. Architect 不再套用市场、竞争、政策等固定行业报告模板，而是围绕技术问题生成 3 个针对性章节。
2. 深层检索和补充检索继承 `search_web`、`search_local` 与 `kb_name`，本地模式不会误调用外部搜索接口。
3. 多个知识库切片来自同一文档时，事实仍能正确绑定唯一来源。
4. 只有研究提纲明确需要结构化数据或图表时，才运行数据分析和代码绘图 Agent。
5. Writer 使用来源白名单，移除未经检索的链接和图表，并确定性补充真实证据来源。
6. Critic 的严重问题具有确定性否决权，避免模型同时输出“存在严重问题”和“审核通过”。
7. 运行时队列不会写入检查点；评审路由、最终报告、质量分和未解决问题均可保存和恢复。

## 自动化与集成验证

- Agent 与来源治理回归测试：15/15 通过。
- PostgreSQL 检查点集成验证：保存、完成状态更新、恢复、运行时字段清理全部通过。
- Docker 基础服务：PostgreSQL、Redis、etcd、MinIO、Milvus 健康。

## 当前边界

现有公开全文语料不足以直接支撑 OpenCAPI、OpenVPX 等具体接口对比及部分工程环境结论。系统现在会识别这一问题并继续检索，但若知识库中没有相应权威资料，循环检索也无法创造证据。

下一阶段应扩充四个产业方向的高质量公开资料，并建立一组可重复运行的半导体研究评测问题。优先补充“封装与测试”中 Chiplet 接口、互操作性、DFT、热与可靠性方面的官方或标准机构公开材料。
