# 多源行业研究平台架构与验收路线

## 1. 正式定位

本项目不是只回答半导体 PDF 的问答应用，而是一个**证据驱动行业研究平台**。平台提供可复用的
研究工具、Agent 状态机和报告交付能力；半导体全产业链是当前完成深度语料与评测闭环的第一个
垂直领域。

一句话边界：

> 通用平台已经具备多源能力入口，半导体文档证据链已经深度验证；先进封装冻结脱敏
> fixture 已通过 12/12 确定性多源端到端门禁，真实在线多源数据仍需独立验收。

## 2. 平台层与垂直层

### 通用平台层

- 会话、认证、研究记忆和知识库管理；
- 文档 RAG、Web 搜索、新闻政策、招投标、产业数据库、Text2SQL 和公司行情适配器；
- 研究计划、工具调用、写作、审核、取消、恢复和超时；
- SSE 事件、健康检查、Prometheus 指标、容器部署和 CI。

### 半导体垂直层

- 芯片设计与 EDA/IP、材料与设备、晶圆制造、封装测试四个领域配置；
- 领域搜索词、新闻词、招投标词和推荐研究问题；
- 公开来源审核、许可判断、SHA-256、PDF 规范化与四个知识库；
- 术语扩展、混合检索、相邻块补全和半导体专项评测集；
- 可复现的检索、回答、拒答、并发和 Agent 可靠性报告。

新增一个行业时，应复用平台层，只新增领域配置、审核语料、数据映射和独立评测集。

## 5. 目标联合研究场景

建议只做一个足够深入的黄金路径：

> 结合权威技术资料、近期政策、设备招投标和上市公司数据，分析先进封装设备国产化机会、
> 主要受益环节与关键风险。

期望流程：

1. Architect 将问题拆成技术壁垒、政策环境、采购需求、公司表现和风险；
2. RAG 提供先进封装、互连、测试与设备环节的权威技术证据；
3. 新闻/政策工具返回有发布时间和发布方的时效性事件；
4. 招投标工具汇总采购方、设备类别、地区、金额和公告状态；
5. Text2SQL 对结构化行业/公司数据执行只读查询并返回行级依据；
6. 股票工具提供带 `as_of` 时间的市场信号，明确它不是基本面结论；
7. Writer 区分事实、跨源推断、风险和待验证项；
8. Critic 检查每类结论是否由对应类型的证据支持。

## 6. 统一证据契约

PDF 页码不能表达 SQL 行、新闻发布时间或行情时点。下一阶段应把所有工具输出归一为统一证据对象：

```json
{
  "evidence_id": "stable-id",
  "source_kind": "document|news|policy|bidding|sql_row|market_quote",
  "title": "人类可读标题",
  "publisher": "发布方或数据提供方",
  "url": "可选原始链接",
  "published_at": "可选发布时间",
  "retrieved_at": "采集时间",
  "as_of": "行情或统计口径时点",
  "locator": {"page": 12, "row_ids": [], "notice_id": null},
  "content": "支持结论的最小证据内容",
  "content_hash": "sha256",
  "quality_tier": "official|primary|secondary|unknown",
  "license_or_terms": "许可或使用条款判断"
}
```

目标不是让所有来源长得一样，而是让 Writer、Critic、前端引用和评测器都能回答：这条结论来自
哪里、什么时间、哪个定位、是否允许使用、用户如何复核。

## 7. 多源专项评测

首版已实现 12 题，规模小但逐题具有机器可读门禁：

- 3 题需要文档 + 新闻/政策；
- 3 题需要文档 + 招投标；
- 2 题需要文档 + Text2SQL；
- 2 题需要文档 + 公司/行情；
- 2 题故意缺少关键来源，必须拒绝下确定性结论。

每题至少检查：

- 工具选择是否正确；
- 必需来源类型是否都被召回；
- 数字是否带口径和时点；
- 引用定位是否可复核；
- 推断是否明确标注；
- 来源冲突是否被披露；
- 关键证据缺失时是否拒绝或降级。
