# 证据驱动行业研究平台：半导体垂直闭环与学习路线

## 1. 北极星目标

面向产业研究员、企业战略人员与技术人员，建设可复用的多源行业研究平台；以芯片设计、
半导体材料与设备、晶圆制造、封装测试四个环节作为第一个深度垂直领域，从可授权、可追溯的
专业资料、公开情报和结构化数据中收集证据，生成带有效引用的研究结论；当证据不足时明确拒答
或标注不确定性。多源平台目标和验收顺序见 `docs/MULTI_SOURCE_RESEARCH_PLATFORM.md`。

项目完成的标准不是“页面可以聊天”，而是：

1. 任一关键结论都能追溯到原始资料和具体证据片段；
2. 数据入库、检索、生成、引用和拒答均有独立且可重复运行的评测；
3. 新环境可以依据文档一键启动，并复现公开评测结果；
4. 对失败案例、性能瓶颈和系统边界有可审计记录；
5. 项目负责人能够解释关键实现、设计取舍、替代方案和常见故障。

## 2. 项目边界

### 2.1 核心用户任务

- 单文档事实查询：从标准或技术报告中定位参数、定义和结论；
- 跨段落综合：组合一份或多份资料中的互补证据；
- 跨产业链研究：分析一项技术变化对设计、设备材料、制造和封测环节的传导；
- 来源核验：查看结论由哪份资料、哪个片段支撑；
- 知识边界判断：资料不足、型号不存在或问题超出知识库时安全拒答；
- 深度研究：规划研究问题、收集证据、分析、写作、审核和修订。

### 2.2 暂不作为核心目标

- 实时股票交易建议或自动执行生产操作；
- 用少量公开资料训练具备完整半导体知识的基础模型；
- 通过增加固定题目规则追求评测集满分；
- 在没有领域专家复核时声称达到真实业务准确率。

## 3. 端到端业务闭环

```text
可信资料发现与授权
        ↓
原始文件留存、元数据与哈希登记
        ↓
PDF 解析、结构恢复、清洗与质量检查
        ↓
分块、Embedding、索引与可恢复入库
        ↓
查询理解、混合召回、重排与上下文预算
        ↓
基于证据生成、引用绑定与无证据拒答
        ↓
多阶段研究、审核、修订与检查点恢复
        ↓
离线评测、在线观测、失败归因与迭代
```

每一层必须同时具备：输入输出契约、失败处理、日志、测试和可解释指标。

## 4. Definition of Done

### Gate A：资料治理

- 每份资料保存来源 URL、发布方、发布日期、版本、许可策略、内容哈希和抓取时间；
- raw、normalized 与索引记录可以通过稳定 ID 对齐；
- 未批准全文入库的资料不能绕过治理规则；
- 重复文档、版本更新和失效链接有明确处理策略。

### Gate B：文档解析与入库

- 公开基准语料可以从空库重复完成解析、分块、向量化和入库；
- 解析结果保留标题层级、页码或等价定位信息；
- 嵌入批处理支持重试、数量校验和断点恢复；
- 不允许空向量、残缺文档或失败状态静默进入 completed；
- 输出解析成功率、页数、块数、失败原因和耗时分布。

### Gate C：检索

- 分别评测向量召回、词法召回、混合检索、重排、邻接扩展和查询改写；
- 报告 Recall@K、MRR/nDCG、金标来源命中率、无证据拒绝率和上下文块数；
- 必须保留 ablation，证明每个策略的真实增益与额外开销；
- 不得根据 test 或 hidden 单题增加特判。

### Gate D：回答与引用

- 回答中的关键事实能够映射到具体证据；
- 分开统计事实覆盖、引用合法性、引用正确性、引用完整性和拒答正确率；
- 检索失败、模型失败和引用失败能够被单独归因；
- 高风险生产操作只提供分析与核验建议，不生成未经授权的执行承诺。

### Gate E：研究 Agent

- 规划、检索、分析、写作、审核、修订均有结构化状态和输入输出；
- 最大迭代次数、取消、超时、失败降级和检查点恢复可以测试；
- 严重证据问题对最终通过具有确定性否决权；
- 准确描述实际运行路径，不能把未启用的框架能力写成线上能力。

### Gate F：工程质量

- 后端单元测试、集成测试和前端 lint/build 在 CI 中通过；
- 依赖可锁定，环境变量经过校验，仓库不包含密钥和运行时数据库；
- 一条命令启动依赖与应用，一条命令运行公开 smoke test；
- API、数据库迁移、健康检查和清理流程有文档；
- 代码归属、许可证和可公开范围经过确认。

### Gate G：性能与可观测性

- 记录解析、Embedding、召回、重排、首 token、生成和总耗时；
- 报告冷启动与预热后的平均值、P50、P95、P99；
- 记录 token、模型调用、缓存命中和失败重试；
- 完成单用户、并发和长文档三类压力测试；
- 性能阈值失败不能通过放宽 SLA 隐藏。

### Gate H：作品集与复现

- README 先讲业务问题、架构、差异化与可核验结果，再讲启动步骤；
- 提供一个跨产业链案例、一个无证据拒答案例和一个失败案例；
- 所有简历数字都对应仓库命令和机器可读报告；
- 私有评测答案、公司资料和未经授权代码不进入公开仓库。

## 5. 当前基线（2026-07-17）

### 已有能力

- 覆盖四个半导体产业方向的知识库与研究工作流；
- 研究运行、阶段、Agent、LLM、checkpoint、取消与 Redis 锁已暴露低基数 Prometheus 指标，
  可选 Compose profile 包含 7 天存储与四条可靠性告警；
- 12 份 PDF、1327 页完成 v2 规范化，失败 0；
- 15 份获批资料完成无密钥全语料 Milvus Lite smoke：36 个文档领域任务、
  5256 个切片、4 个 collection 对账通过、0 重复 chunk ID；
- 80 题分为 regression、development、test、hidden 四层；
- 检索严格通过：regression 20/20、development 20/20、test 15/20、hidden 18/20；
- regression 历史宽松回答门槛为 20/20；新增逐概念“回答句—引用—切片”
  对齐后严格质量为 16/20，无证据拒答 4/4，引用支持达标 12/16；
- 结构化逐论断校验已实现“模型提议—服务端校验—服务端渲染—失败关闭”，
  但同集实验仅 12/20、SLA 19/20，因此作为实验开关保留且默认关闭；
- 新严格回答评测平均延迟 6.385 秒，P95 12.477 秒，20 秒 SLA 为 20/20；
- 同步 RAG 链路已从 Uvicorn 事件循环移入 StreamingResponse 线程池；
  受控用例在并发 4 时 8/8 质量通过、P95 10.217 秒，并发 8 时
  16/16 通过但 P95 升至 19.919 秒且吞吐几乎不再增长；
- 600,400-token 合成长文档集已验证证据预算会被确定性截断到
  6,000 token 以内，且重排异常不再绕过该边界；
- development 检索消融表明：单查询 dense 3/20，多查询 dense 7/20，
  混合多查询 20/20；后者 MRR 0.7083、nDCG@3 0.7185、P95 2.789 秒；
- 前端可以完成生产构建。

### 指标解释边界

- hidden 18/20 是历史检索严格通过率，不是端到端回答准确率；
- regression 20/20 不能外推为真实业务准确率 100%；
- 当前端到端回答结果只覆盖 regression，不代表 test/hidden 回答质量；
- 现有 test/hidden 标签曾经由聚合数据和生成脚本暴露，已有分数只能作为
  受污染的阶段验收，不是盲测；
- 后续大改应由独立复核者在代码冻结后新建 blind-v2，问题和答案都不对调参者可见；
- 当前公开语料规模仍小，系统能力不能表述为覆盖全部半导体知识。

## 7. 实施里程碑

### M0：冻结基线与合规边界

- 确认代码与数据归属、公开范围和个人贡献；
- 初始化版本控制，保护私有答案、密钥、数据库和构建产物；
- 记录现有测试、构建和评测基线，不修改历史报告。

### M1：可复现工程基线

- 建立 Python 虚拟环境与锁定依赖；
- 统一后端测试入口，修复导入副作用；
- 清零前端 lint，保留 build 通过；
- 增加 CI、健康检查与公开 smoke test。

### M2：数据与检索闭环

- 从空库复现公开资料入库；
- 建立检索组件 ablation 和按领域/题型失败分析；
- 解决通用的实体归一化、跨段证据和重排问题；
- 冻结策略后运行新的 test/blind 验收。

### M3：回答、引用与 Agent 闭环

- 建立 claim-evidence 对齐评测；
- 在非 regression 集运行端到端回答评测；
- 验证取消、超时、恢复、审核否决和降级；
- 为完整研究报告增加来源覆盖和不确定性检查。

### M4：性能、部署与作品集

- 增加分阶段 tracing、缓存和模型预热；
- 完成并发与长文档压测；
- 更新 README、架构图、演示案例和复现命令；
- 生成只包含可核验事实的简历表述。

## 8. 深入学习标准

每个模块都按“能解释、能实现、能诊断”三级掌握。

| 模块 | 能解释 | 能实现 | 能诊断 |
| --- | --- | --- | --- |
| PDF 解析 | 版面、内容流、OCR 与结构恢复差异 | 页眉页脚清理、标题恢复、稳定分块 | 乱码、断词、表格、跨页和块爆炸 |
| Embedding | 语义空间、维度、归一化和距离度量 | 批处理、重试、缓存、断点恢复 | 模型/维度不一致、空向量、吞吐瓶颈 |
| 检索 | dense、sparse、hybrid、rerank | 融合、Top-K、邻接扩展、上下文预算 | 漏召回、错排序、术语漂移、跨段缺失 |
| RAG 生成 | grounded generation 与上下文约束 | 引用编号、拒答、证据压缩 | 幻觉、引用错位、lost-in-the-middle |
| 评测 | Recall@K、MRR、nDCG、faithfulness | 分层数据集、金标、ablation | 泄漏、过拟合、指标与业务错位 |
| Agent | 状态机、工具调用、反思与检查点 | 规划、循环、取消、恢复、审核 | 死循环、状态污染、无效反思、成本失控 |
| 服务工程 | SSE、数据库、缓存和并发 | API、迁移、健康检查、CI | 超时、连接泄漏、冷启动、并发退化 |

每完成一个里程碑，应产出：

1. 一页架构与数据流说明；
2. 一份关键代码导读；
3. 一份方案选择与替代方案对比；
4. 至少三个真实失败案例及根因；
5. 十个由浅入深的面试追问及答案；
6. 一条可复现实验命令和对应报告。

## 9. 面试深挖验收问题

项目负责人至少应能回答：

1. 为什么选择混合检索，向量权重如何确定，是否做过 ablation？
2. 相邻块扩展解决什么问题，为什么不会跨文档污染证据？
3. 如何判断引用“合法”但并不真正支撑结论？
4. 为什么 test 召回 16/16，严格通过却只有 15/20？
5. hidden 为什么只能阶段性运行，运行后为何需要新建 blind-v2？
6. PDF 从 5559 块恢复为 1501 块说明了什么，如何证明不是丢内容？
7. 多查询改写为什么可能提高召回却降低精度或增加延迟？
8. 为什么不能把回归集 20/20 写成业务准确率 100%？
9. Agent 审核循环什么时候有价值，什么时候只是增加 token 消耗？
10. 系统在 Milvus、Embedding 服务或 LLM 不可用时如何降级与恢复？
11. 如何区分检索失败、生成失败和引用绑定失败？
12. 如果语料扩大一百倍，当前词法扫描、缓存和上下文策略哪里先成为瓶颈？

## 10. 执行原则

- 先冻结基线，再修改，再运行未见测试；
- 每次优化必须对应明确失败类型和可验证指标；
- 不为单题写规则，不删除失败案例，不篡改历史报告；
- 优先修正确性与可复现性，再增加功能和视觉效果；
- README、简历和面试中的表述必须严格小于或等于证据能够支持的范围。
