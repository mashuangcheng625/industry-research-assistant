# 项目介绍、简历与演示手册

本文用于把工程事实转化为招聘沟通材料。所有数字都必须能由代码、固定数据集或
`reports/` 中的报告复核；没有独立证据的能力不写成完成态。

## 1. 30 秒项目介绍

我搭建了一个证据驱动行业研究平台，并以半导体全产业链作为第一个深度垂直落地。平台接入
专业知识库、新闻政策、招投标、产业数据库、Text2SQL 和公司行情；半导体垂直层进一步打通
来源审核、PDF 解析、混合检索、带页码引用、无证据拒答、离线评测、Agent 取消恢复和容器部署。
当前半导体语料有 12 份真实 PDF、1,327 页和 5,256 个块；
在固定 regression 集上，混合检索严格通过 20/20，端到端回答严格质量通过 16/20，
4 个无证据问题全部拒答。多源工具已接入，并完成冻结 fixture 的 12 题联合研究门禁；
真实在线数据仍缺少同等级盲测，所以我会明确区分“可重复 fixture 闭环”和“线上质量验证”。

## 2. 2 分钟项目介绍

产业研究的核心痛点不是缺少语言流畅的摘要，而是权威文档、时效新闻、采购需求和结构化数据
分散在不同系统里，来源质量、时间口径和结论复核都很困难。因此我采用“通用平台 + 垂直领域”
的双层设计，而不是做一个只面向单批 PDF 的聊天机器人。

通用平台层负责会话、知识库、新闻政策、招投标、Text2SQL、股票信号和 Research Agent；
领域层负责术语、来源政策、知识库和评测集。半导体是当前完成深度闭环的领域，其他多源模块
已具备接口、页面、统一 Evidence 契约和 12 题 fixture 门禁，但真实在线数据仍需独立盲测，
`CitationLocator` 已作为兼容字段接入 Writer、Chat/Research API 与前端来源卡片。

数据侧，我建立来源注册表，记录发布机构、URL、许可判断、审核状态和 SHA-256；只允许批准资料
进入解析与索引，并审计 PostgreSQL 和 Milvus 的一致性。检索侧，我先做了消融：单查询稠密
检索只通过 3/20，多查询稠密是 7/20，引入词法信号和多查询后达到 20/20，再用相邻块补回
跨段上下文。生成侧，模型只能使用预算内证据，回答带文档与页码引用；评测同时检查答案要点、
引用来源、逐主张支持和无证据拒答。

工程侧，我修复了同步 RAG 阻塞异步事件循环的问题，把流式生成交给线程池迭代；并发 4 的
8 次请求全部通过，P95 为 10.217 秒。研究 Agent 支持超时、取消、检查点和精确恢复，审核中的
critical 问题不会因为达到最大迭代次数而被错误放行。最后用 Docker Compose、readiness、
Prometheus 告警和 CI 检查形成可复现闭环。

我最重视的一点是证据边界：20/20 只是固定回归集，不等于真实业务 100%；本地 4B 语义裁判
效果不稳定所以默认关闭；当前在线编排也不宣称由 LangGraph 执行。

## 3. 简历表述

### 偏大模型应用 / RAG 岗

- 设计“通用行业研究平台 + 半导体垂直层”架构，接入文档 RAG、新闻政策、招投标、Text2SQL
  和公司行情，并以独立领域配置、来源注册表和评测集隔离通用编排与行业知识。
- 设计并实现半导体全产业链证据驱动研究助手，贯通 17 个候选来源的审核与哈希治理、
  12 份真实 PDF（1,327 页）解析、5,256 个证据块索引、带页码引用生成和无证据拒答。
- 基于 40 题分层评测集完成检索消融，将严格检索通过从单查询 dense 的 3/20 提升至
  混合多查询的 20/20；在 regression 端到端评测中取得 16/20 严格质量通过和 4/4 正确拒答。
- 建立逐主张引用评测、questions-only 测试集隔离和失败关闭语义校验，明确区分召回成功、
  引用出现与证据真正支持结论，降低幻觉和评测泄漏风险。

### 偏大模型平台 / 后端岗

- 为研究 Agent 实现超时、取消、检查点、精确恢复和审核门禁，修复迭代上限错误放行、
  伪恢复、异常吞没和同 session 并发覆盖等状态一致性问题。
- 定位同步 RAG 调用阻塞 Uvicorn 事件循环的性能瓶颈，将流式响应改为线程池迭代；健康检查
  延迟由 2.753 秒降至 0.081 秒，并完成并发 1/2/4/8 的质量门控容量测试。
- 构建非 root 后端与 Nginx 前端镜像，配置模型感知 readiness、Prometheus 多 worker 指标、
  4 条告警和 Grafana 看板，以 457 个后端测试（453 unit + 4 integration）、
  前端 lint/build、Compose 和数据契约校验作为统一工程门禁。

不要把两组全部堆进一页简历。根据岗位选择三条；多源模块使用“冻结 fixture 端到端门禁”而不是
“全面打通”。每条都准备好“为什么这样设计、代码在哪里、
指标如何算、失败案例是什么”的追问答案。

## 4. 三个 STAR 深挖案例

### 案例 A：检索从 3/20 到 20/20

**Situation**：行业术语有中英文、缩写和上下游表达差异，单一向量查询漏召回严重。

**Task**：提高关键来源进入 top-k 的概率，同时保留可解释的对照实验。

**Action**：建立按产业环节平衡的 development 集；固定其余变量，对比单查询 dense、
多查询 dense、混合多查询；加入词法命中和相邻块补全，并保存每题排名与耗时。

**Result**：严格通过依次为 3/20、7/20、20/20；混合方案 MRR 0.7083、nDCG@3 0.7185、
P95 2.789 秒。后续没有把 `rrf_score` 诊断字段说成线上最终 RRF 排序。

可能追问：词法分为何有效？多查询会不会引入噪声？为什么不用 cross-encoder？20/20 是否过拟合？

### 案例 B：Agent 恢复不是“重新跑一遍”

**Situation**：旧实现保存了 checkpoint，却无法证明恢复后从正确阶段继续；达到迭代上限时，
还可能把含 critical 问题的报告标记为完成。

**Task**：让取消、失败、恢复和审核结果具有明确状态语义。

**Action**：保存 `last_completed_phase` 和可序列化状态；恢复时根据阶段游标进入下一节点；
为取消、超时、同 session 互斥和审核严重级别建立测试；所有未解决 critical/major 问题都进入失败态。

**Result**：真实本地模型完成取消—恢复控制面验证；一次完整审核得到 quality 4.0、critical 1、
major 2，并正确结束为 `research_review_failed`，没有错误放行。

可能追问：checkpoint 原子性怎么保证？进程在写入中途崩溃怎么办？幂等边界在哪里？

### 案例 C：流式接口为什么拖慢健康检查

**Situation**：聊天接口虽然定义为 async generator，但内部检索和模型 SDK 都是同步调用，
一个生成请求会让同一 Uvicorn 事件循环上的健康检查延迟到 2.753 秒。

**Task**：保持 SSE 协议不变，消除事件循环阻塞，并测出本地模型容量拐点。

**Action**：把三个响应生成器改为同步 generator，让 Starlette 在线程池中迭代；增加带答案质量
门槛的并发脚本，分别测并发 1、2、4、8，并保留一次错误选题导致质量下降的无效实验作为反例。

**Result**：健康检查降至 0.081 秒；并发 4 时 P95 10.217 秒，并发 8 时升至 19.919 秒，
因此只把并发 4 附近描述为本地 4B 模型的实验容量，不作生产承诺。

可能追问：线程池大小如何决定？GPU 推理能否真正并行？为什么不用异步 SDK 或任务队列？

## 5. 15 分钟演示脚本

### 0–2 分钟：问题与证据链

打开 README 的双层架构图，先说明通用平台负责多源工具和 Agent，半导体层负责领域知识与评测。
再用一句话解释“证据驱动”：每个可核查结论都能回到具体来源、时间和定位，找不到足够证据时
系统不猜。

### 2–5 分钟：资料治理与入库

展示来源注册表中的 approved 和 metadata-only 条目，再展示一个规范化块的 `source_id`、页码、
文档哈希。运行来源校验或展示最新入库审计报告。

### 5–9 分钟：三个问答

运行 `make demo-rag`：

1. 一个设计/EDA 正例，展示术语扩展和来源页码；
2. 一个先进封装跨段问题，展示相邻块补全；
3. 一个语料外问题，展示拒答而不是编造引用。

回答后主动点击或核对引用原文，不只展示生成文本。

### 9–12 分钟：消融与失败案例

展示 3/20 → 7/20 → 20/20 的检索消融；再展示端到端只有 16/20，解释“召回到”不等于
“回答中的每个主张都被正确支持”。选一个失败样例说明下一步如何改，而不是只展示满分结果。

### 12–14 分钟：可靠性与性能

展示 Agent 取消—恢复状态游标，以及 critical 审核不放行；展示并发 4 与并发 8 的 P95，
说明容量拐点和线程池修复。

### 14–15 分钟：边界与下一步

明确当前不是生产系统，新闻、招投标、Text2SQL 和股票是已接入但未完成同等级在线盲测的
平台能力。多源证据契约、先进封装 12 题 fixture 和 blind-v2 首次审计已经完成；下一步优先
制作经过来源预校验的 blind-v3，并推进可扩展词法索引与连续数据库迁移验收，
再推进可扩展词法索引和连续数据库迁移验收。

## 7. 个人贡献与公开前检查

只有实际由你完成、理解并能定位代码的内容才写入个人简历。面试前逐项确认：

- 能从请求入口讲到检索、上下文预算、模型调用、引用校验和 SSE 输出；
- 能手算一个检索得分，并解释每个权重和 top-k 的作用；
- 能从 JSON 报告复算通过率、MRR、nDCG 和 P95；
- 能画出 Agent 状态转移并解释取消、恢复、超时和审核失败；
- 能复现至少一个成功实验和一个失败实验；
- 能区分自己原始实现、后续修复、第三方框架能力和模型本身能力；
- 已获得公司代码、数据、原始报告和公开资料再分发的书面授权。
