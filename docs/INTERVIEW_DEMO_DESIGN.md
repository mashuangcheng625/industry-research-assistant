# 5 分钟面试演示模式设计

## 1. 目标

这条演示路径不试图在几分钟内展示平台的所有页面，而是用一个有证据问题、
一个无证据问题和一组可复核的工程指标，让面试官在 5 分钟内看到：

1. `text-embedding-v4` 云端索引与 `bge-m3` 本地索引的双路检索；
2. RRF 融合、`qwen3-rerank` 精排和路由降级状态；
3. 回答中的结论可以回到原始文档、页码和证据片段；
4. 没有足够证据时系统会拒答，不使用模型常识补全虚构参数；
5. 上述能力有真实模型冒烟、固定评测、自动化测试和 CI 证据。

演示页是现有产品的精简入口，不是第二套业务系统，也不引入专用的 Mock
回答链路。

## 2. 范围与非目标

### 2.1 演示范围

- 一个先进封装正例；
- 一个虚构型号拒答反例；
- 检索路由、重排、引用和降级状态展示；
- 固定报告、测试数量和 CI 结果的可核对入口；
- 可选的 Research Agent 取消、检查点和恢复说明。

### 2.2 非目标

- 不在主演示中逐一打开新闻、招投标、Text2SQL、股票和长期记忆页面；
- 不现场下载 PDF、审核来源或全量重建向量索引；
- 不把历史报告伪装成当次实时结果；
- 不为了演示绕过 JWT、证据门禁或模型失败逻辑；
- 不宣称固定评测集的结果等同于开放域准确率。

## 3. 用户与运行前置条件

演示使用现有 `research_demo` 用户与 JWT 鉴权。演示前完成以下准备，
现场不再执行入库操作：

- FastAPI、PostgreSQL、Redis 和 Milvus readiness 通过；
- Ollama 已加载 `industry-qwen3:4b` 与 `bge-m3`；
- 百炼 `DASHSCOPE_API_KEY` 可用；
- 四个半导体知识库已建立；
- 演示语料已建立云端和本地两套向量 Collection；
- 演示用户对目标知识库和文档有访问权限。

页面打开时应进行只读预检，但不自动发起付费模型调用。预检显示：

```text
后端 API       就绪 / 失败原因
PostgreSQL       就绪 / 失败原因
Redis            就绪 / 失败原因
Milvus           就绪 / 失败原因
Cloud Embedding  已配置 / 未配置
Local Embedding  已配置 / 未配置
Rerank           已配置 / 未配置
知识库双索引   就绪 / 缺失的路由
```

## 4. 页面信息架构

建议新增需要登录的 `/demo` 路由，不修改现有聊天页的主要交互。

```text
┌─────────────────────────────────────────────────────────────┐
│ 半导体研究助手 · 面试演示      服务正常  Hybrid  无降级 │
├──────────────┬─────────────────────────┬─────────────────────┐
│ 演示场景     │ 问题与回答              │ 证据链              │
│              │                         │                     │
│ ① UCIe 正例  │ 预置问题                │ text-v4 召回        │
│ ② 无证据拒答 │ 流式回答                │ bge-m3 召回         │
│ ③ Agent 恢复 │ 可点击引用 [1][2][3]      │ RRF → Rerank        │
│              │                         │ 文档名、页码、分数  │
├──────────────┴─────────────────────────┴─────────────────────┤
│ 459 单测 ｜ 10 集成测试 ｜ Hybrid 5/5 ｜ CI Passed         │
└─────────────────────────────────────────────────────────────┘
```

左侧场景卡只将问题填入输入框，不自动发起请求。用户必须点击
“开始演示”后才调用真实检索和模型服务。

## 5. 场景一：Hybrid 证据问答

### 5.1 固定问题

使用 regression 集中的 `packaging-ucie-018`：

> UCIe 等 die-to-die 互连标准为什么对不同厂商芯粒的互操作重要？

### 5.2 必须展示的现场信息

- 本次查询的 Embedding 模式为 `hybrid`；
- 云端路由为 `text-embedding-v4`；
- 本地路由为 `bge-m3`；
- 两路返回的候选数量；
- 融合后候选数量与 Rerank 模型；
- 每条最终证据的 `retrieval_routes`、重排分数和降级标记；
- 文档名、页码、切片序号和可点击原文；
- 回答中的引用编号与证据片段一一对应。

右侧证据卡的最小表达：

```text
[1] 文档名称 · 第 42 页
    routes: cloud + local
    rerank: 0.91
    degraded: none

[2] 文档名称 · 第 45 页
    routes: cloud + local
    rerank: 0.84
    degraded: none
```

点击回答引用后，打开现有证据切片抽屉，定位到对应文档、页码和原文，
不只展示模型生成的结论。

### 5.3 成功标准

- 回答请求成功完成；
- 最终证据中至少一条同时包含 `cloud` 和 `local`；
- 没有 `degraded_route`；
- 回答至少有一个可点击的有效引用；
- 引用打开后可以看到非空原文和定位信息。

## 6. 场景二：无证据拒答

### 6.1 固定问题

使用 regression 集中的 `packaging-negative-020`：

> 请给出虚构 NX-999 量子芯粒混合键合的官方焊盘尺寸和量产温度窗口。

### 6.2 必须展示的决策信息

```text
严格标识符：NX-999
支持该参数的证据：0
证据门禁：未通过
决策：拒答
```

返回内容必须明确表达“知识库中没有足够证据”，不得给出具体焊盘尺寸、
温度或伪造的文档引用。

### 6.3 成功标准

- 返回确定性拒答或明确的证据不足说明；
- 不包含任何被当作事实的 NX-999 具体参数；
- 不生成越界引用；
- 页面将拒答归因于证据门禁，而不是笼统显示“模型不知道”。

## 7. 场景三：Research Agent 可靠性（追问备用）

这一场景默认不在 5 分钟主演示中发起完整深度研究，而是用于回答
面试官对 Agent 状态一致性的追问。

展示一条已保存的状态链：

```text
Planning     completed
Researching  completed
Analyzing    completed
Writing      completed
Reviewing    failed: critical finding
Completed    false
```

可展示的两个动作：

1. **查看检查点**：展示 `last_completed_phase`、审核结果和下一可执行阶段；
2. **从检查点恢复**：证明恢复从正确阶段继续，不是从 Planning 重新执行。

审核中存在未解决的 `critical` 或 `major` 问题时，页面不得显示为完成。

## 8. 5 分钟现场脚本

| 时间 | 操作 | 讲解重点 |
| --- | --- | --- |
| 0:00–0:30 | 打开演示页，确认预检全绿 | “这是证据驱动研究平台，不是普通聊天机器人。” |
| 0:30–2:30 | 运行 UCIe 正例 | 解释双 Collection、RRF、Rerank 和无降级状态 |
| 2:30–3:20 | 点击一个回答引用 | 展示文档名、页码、原文和回答论断的对应关系 |
| 3:20–4:10 | 运行 NX-999 反例 | 展示严格标识符与证据不足拒答 |
| 4:10–4:40 | 打开验证栏 | 展示 459 项单测、10 项集成测试、Hybrid 5/5 和 CI |
| 4:40–5:00 | 说明能力边界 | 固定集结果不代表开放域 100%，当前定位是研究型 MVP |

### 8.1 开场话术

> 我没有把重点放在模型能不能生成流畅文字，而是解决行业研究中结论是否有来源、
> 能否定位，以及证据不足时是否拒答的问题。下面用一个正例和一个反例展示完整链路。

### 8.2 结束话术

> 项目的核心工程价值是把文档治理、双路检索、重排、引用、拒答和评测连成可复核闭环。
> 当前固定 regression 检索集是 20/20，但端到端严格质量是 16/20，所以我不会把它描述成
> 开放域准确率 100% 或已上线的生产系统。

## 9. 工程证据栏

底部验证栏的数字不得写死为不可核对的宣传语。每项都应能打开对应报告或 CI：

| 展示项 | 当前口径 | 证据 |
| --- | ---: | --- |
| 单元测试 | 459 项 | `make test-backend-unit` |
| Milvus Lite / Redis 限流与持久任务 / Alembic / PostgreSQL 备份恢复集成测试 | 10 项 | `make test-backend-integration` |
| 真实模型 Hybrid 冒烟 | 5/5 | `reports/p1-5-model-smoke.json` |
| regression 检索 | 20/20 | baseline manifest 指向的冻结报告 |
| regression 端到端严格质量 | 16/20 | baseline manifest 指向的冻结报告 |
| 无证据拒答 | 4/4 | baseline manifest 指向的冻结报告 |
| 持续集成 | Passed | GitHub Actions 当前 `main` 最新提交 |

如果报告无法读取或口径与代码不一致，页面应显示“待核对”，不应继续展示过期数字。

## 10. 失败和降级展示

现场失败也必须保持真实语义：

| 故障 | 页面行为 |
| --- | --- |
| 云端 Embedding 失败 | 显示 `cloud` 降级，不显示“Hybrid 成功” |
| 本地 Embedding 失败 | 显示 `local` 降级，不隐藏 Ollama 不可用 |
| Rerank 失败 | 标记保留原检索分数的降级排序 |
| 生成模型失败 | 保留已检索证据，显示模型错误而不伪造回答 |
| 引用定位失败 | 引用不可点击并标记 locator 缺失 |
| 知识库未就绪 | 阻止开始演示，列出缺失的 Collection |

页面可提供“查看最近一次冻结报告”作为备用，但必须显示报告时间和
“历史实测”标签，不得与当次实时结果混淆。

## 11. 实现分期

### Phase A：无新页面的可执行演示

- 保留现有 `make demo-rag`；
- 固定运行 `design-flow-001`、`packaging-ucie-018` 和
  `packaging-negative-020`；
- 在现有聊天页手动打开证据抽屉；
- 以本文第 8 节作为现场话术和时间控制。

### Phase B：`/demo` 专用页

- 增加三个场景卡和一键填入问题；
- 复用现有聊天 SSE 和证据抽屉；
- 增加 Hybrid 路由、Rerank 和降级可视化；
- 增加服务与双索引预检；
- 增加工程证据栏与报告链接。

### Phase C：面试追问面板

- 展示 Agent 状态迁移、取消和检查点；
- 展示一个审核拒绝放行的失败案例；
- 展示检索消融 `3/20 → 7/20 → 20/20`；
- 展示并发 4 与并发 8 的容量拐点。

Phase C 不应阻塞 Phase B 的交付。

## 12. 验收合同

### 12.1 路由与权限

- `/demo` 只对已登录用户可见；
- 未登录访问使用现有鉴权流程；
- 不新增跳过 JWT 的演示后门；
- 只读预检不消耗云端生成或 Embedding 额度。

### 12.2 功能和状态

- 正例、拒答与 Agent 备用场景都有明确状态；
- 流式中、完成、无证据、任一路降级、模型失败和知识库未就绪都可见；
- 连续点击不会发起重复研究任务；
- 切换场景不污染上一场景的证据和状态。

### 12.3 行为保持

- 不修改现有聊天、知识库和 Research Agent API 契约；
- 不改变模型路由、检索排序、拒答和引用判定逻辑；
- 不使用写死答案、写死分数或伪造的成功状态；
- 移动端可以纵向浏览三个区域，但面试验收的主视口为 1440×900。

### 12.4 质量门禁

- 前端 `npm run lint` 通过；
- 前端 `npm run build` 通过；
- 后端现有自动化测试不回归；
- 演示正例的引用可定位；
- 演示反例不产生虚构参数；
- 真实 Hybrid 任一路降级时不得显示通过；
- 桌面端 1440×900 在无滚动首屏中可同时看到场景、回答和证据链三个核心区域。

## 13. 面试中的准确表述

| 不准确表述 | 建议表述 |
| --- | --- |
| “混合检索永远不会失败” | “任一路失败会显式降级，真实冒烟门禁要求双路命中” |
| “检索准确率 100%” | “固定 regression 集检索严格通过 20/20” |
| “消除了幻觉” | “通过严格标识符、证据门禁、引用清洗和拒答降低无证据生成” |
| “线上由 LangGraph 执行” | “当前在线主链路是自定义有状态编排” |
| “这是生产级平台” | “这是有可复现工程门禁的研究型 MVP” |
