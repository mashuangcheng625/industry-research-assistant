# RAG 性能、并发与长上下文验收

## 1. 结论

2026-07-17 在同一本地环境、`industry-qwen3:4b` 和两条固定 regression
用例上进行闭环并发测试。修复 Uvicorn 事件循环阻塞后：

- 并发 2 的吞吐从 0.301 提升到 0.371 req/s，平均延迟从 6.637 降至
  4.657 秒；
- 正在处理 RAG 请求时，`/health/live` 从 2.753 秒降至 0.081 秒；
- 并发 4 下 8/8 请求成功且质量门槛通过，吞吐 0.412 req/s，P95 10.217 秒；
- 并发 8 下 16/16 通过，但吞吐仅 0.417 req/s，P95 升至 19.919
  秒，说明本机单模型实例已饱和。

因此，当前本地演示环境的建议并发上限是 4，不是声称系统具备生产容量。

## 2. 根因与修复

`chat_router.py` 中的响应生成器原先定义为 `async def`，但其内部执行的
Embedding、Milvus 检索、重排和 OpenAI-compatible 客户端都是同步阻塞调用。
这会在 Uvicorn 事件循环中执行整条 RAG 链路，使其他请求无法及时调度。

修复方式是将三个 `StreamingResponse` 的内部生成器改为同步生成器。
Starlette 会通过线程池迭代同步 iterable，事件循环因此不再被同步
RAG 链路直接占用。这是 I/O 隔离修复，不会提高单个本地模型的推理吞吐。

## 4. 长上下文和输入边界

`make stress-context-budget` 构造 200 份合成长文档，共 600,400 token。
系统将检索证据确定性限制到 3,002/6,000 token，结果见
`reports/context_budget_stress_2026-07-20.json`。确定性测试还覆盖了：

- 单份文档超预算时跳过它，但仍能保留后续较小证据；
- 重排器异常时只降级排序，不再绕过 token 预算；
- 最多返回 10 份证据，引用 ID 重新连续编号；
- `/chat/completion` 与附件端点拒绝空问题和超过 8,000 字符的问题，
  真实 HTTP 验收均返回 422。

报告同时覆盖 10 条直接模型调用路径、4 档输入压力，共 40 组请求总预算边界：正常请求通过、
临界请求压缩输出、超载请求在 Provider 前拒绝。该试验验证的是预算边界，不是长文档问答质量。
自 2026-07-20 起，主聊天和 Research Agent 已使用默认 32,768-token 总预算计算
系统指令、问题、历史、记忆、证据和输出预留，并保留 6,000-token 证据子上限；
预算不足时会在调用 Provider 前缩小输出上限或确定性拒绝。全部直接模型调用已纳入同一契约。

## 5. 复现

启动本地后端并预热模型后：

```bash
make load-test-chat \
  API_URL=http://127.0.0.1:8000/chat/completion \
  LOAD_CONCURRENCY=4 LOAD_REQUESTS=8 LOAD_WARMUP=1 \
  LOAD_MAX_P95=15 LOAD_MIN_QUALITY_PASS_RATE=1 \
  LOAD_REPORT=reports/chat_load_regression_latest.json

make stress-context-budget
```

`load_test_chat.py` 是闭环负载：每个 worker 等待上一请求完成后再发下一个。
它适合本地容量拐点和回归比较，不代替固定到达率压测、长时稳定性测试或
生产硬件基准。

## 6. 不能外推的结论

- 样本只有两道固定正例，质量 100% 不代表整个 regression 或真实业务准确率；
- 数据来自一台本地机器和一个 4B 模型，不是生产 SLA 或容量承诺；
- 未单独报告首 token 时间：当前服务为了在输出前清洗越界引用，会先缓冲
  完整模型答案；
- 已完成单容器多 worker Prometheus 聚合；尚未完成跨服务 trace、
  GPU/CPU 利用率对齐和持续 soak test；
- 修复前的并发 1 无 warmup，但当时服务已处于预热状态；历史对比不是严格的
  单变量实验，后续应在固定硬件上重新生成标准化基线。
