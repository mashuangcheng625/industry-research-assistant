# 芯片设计 PPA 回归分析演示语料

> 数据声明：本文档是为本地 RAG 功能测试编写的合成案例，不对应任何真实芯片、公司或 EDA 工具结果，不能作为流片依据。

## 1. PPA 回归需要对齐的数据

PPA 分别指性能、功耗和面积。对比两次实现结果时，必须先对齐 RTL commit、综合与布局布线工具版本、时序约束、标准单元库与 PVT corner、寄生参数提取设置、楼层规划以及时钟树策略。如果这些基线不一致，不能把 PPA 变化直接归因于 RTL 修改。

性能侧应检查 WNS、TNS、关键路径组、逻辑级数和高扇出网络；功耗侧应区分动态功耗、漏电功耗，并确认活动率来源；面积侧应对比组合逻辑、寄存器、存储宏和填充单元，同时查看利用率与局部拥塞。

## 2. 合成案例：DSP_CTRL 时序与面积同时退化

演示模块 `DSP_CTRL` 在 RTL commit `demo-b17` 后出现 WNS 从 -0.04 ns 变为 -0.21 ns，标准单元面积增加 8.6%。回归记录显示 EDA 工具、约束、工艺库和 floorplan 均未改变，但该 commit 把一组互斥控制条件改写成多层优先级选择链。关键路径报告中新增了五级选择逻辑，综合结果还复制了部分比较器以处理高扇出。

排查顺序是：首先做基线一致性检查；然后将退化路径映射回 RTL 变更；再对比综合网表的 cell count、逻辑深度和 fanout；最后才评估并行条件计算、预计算选择或流水级调整等方案。未完成功能等价性验证和回归仿真前，不能将建议直接合并到产品分支。

## 3. 检索回答要求

对“如何定位 PPA 退化”的回答应先说明基线对齐，再给出报告到 RTL 的证据链。本案例支持的关键信息是：退化同时体现为 WNS 降低和面积增加；基线工具与约束不变；可疑变更是多层优先级选择链；仍需通过等价性验证和完整回归确认修复。
